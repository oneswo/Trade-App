### ⑦ 代码重构与可维护性
- **状态：** ⬜ 待完成
- **优先级：** 🟡 中
- **目标：** 项目轻量但结构清晰，以后换行业/换类目只需替换内容层，不动架构
- **内容：**

  **后端层**
  - `repository.ts` 体积已较大，按职责拆分为：
    - `src/lib/data/inquiry-repo.ts`
    - `src/lib/data/product-repo.ts`
    - `src/lib/data/admin-repo.ts`
  - API route handler 中的业务逻辑提取为 service 函数（`src/lib/services/`），route 只负责 req/res 转换
  - 统一错误响应格式，避免各路由自行拼 JSON

  **前端层**
  - 将分散在各页面的 `fetch('/api/...')` 调用统一收拢到 `src/lib/api/` 客户端模块（如 `admin-api.ts`、`public-api.ts`）
  - 页面组件只调用封装好的函数，不直接写 fetch
  - 公共 UI 逻辑（如 toast 提示、loading 状态）抽取为可复用 hook

  **内容可替换性**
  - 产品分类、规格字段、导航菜单等"业务配置"统一放入 `src/config/` 目录
  - 换行业时只需修改 `src/config/` 下的配置文件，不需要搜索散落在各组件里的硬编码

  **类型安全**
  - 补全 API 响应类型定义，前后端共用（放 `src/types/`）
  - 消除现有 `any` 类型

- **影响范围：** 全局，不影响功能，只影响代码组织
