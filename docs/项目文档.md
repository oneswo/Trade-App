## 1. 项目概述

面向海外买家的二手工程机械外贸独立站，包含前台展示和后台管理系统。付费投流引流模式。

| 项目 | 说明 |
|------|------|
| 业务类型 | 二手工程机械出口（挖掘机、装载机、推土机等） |
| 目标用户 | 海外买家（非洲、中东、南美、东南亚等） |
| 变现模式 | 询盘留资 → 人工跟单成交 |
| 核心功能 | 产品展示、询盘留言、多语言、管理后台 |
| 部署目标 | Vercel + Cloudflare R2 + Supabase |

## 2. 架构概览

| 组件 | 服务商 | 职责 | 状态 |
|------|--------|------|------|
| 代码部署 | Vercel | Next.js 部署，自动 CI/CD，全球 CDN | ⏳ 待上线 |

| 多语言 | next-intl | 国际化路由与翻译 | ✅ 已接入 |
| 本地开发文件存储 | 本地 `public/uploads/` | 开发阶段临时方案 | ✅ 已实现 |
| 本地开发数据库 | 内存 Mock Store | 开发阶段临时方案 | ✅ 已实现 |

> **本地开发 → 上线切换机制：** 所有基础设施通过环境变量自动切换。
> - `SUPABASE_URL` + `SUPABASE_SERVICE_ROLE_KEY` 同时存在 → 用 Supabase，否则用内存 mock
> - `R2_ACCOUNT_ID` 存在 → 用 Cloudflare R2，否则用本地 `public/uploads/`
> - **不需要改任何代码**

## 3. 流量链路

```
海外用户 → 域名 → Vercel 全球边缘节点 → Next.js 应用
产品图片/视频 → Cloudflare R2（公开访问）
询盘表单提交 → Next.js API Routes → Supabase
管理后台 → /admin/* → Next.js API Routes → Supabase
前台站点设置 → GET /api/site-settings（60s ISR 缓存）→ Supabase
```

> 国内加速备选：如需国内访问加速，可参考 EngVlog 方案接入阿里云 ESA（NS 接入 + 回源 Vercel），无需改动代码。

## 6. 文件存储

### 6.1 本地开发（当前）

上传文件保存至 `public/uploads/{image|video}/{YYYYMMDD}/{timestamp}-{uuid}.{ext}`，通过相对路径 `/uploads/...` 直接访问。

### 6.2 生产环境（R2 适配器已实现）

`src/lib/storage/media-storage.ts` 实现了 `R2MediaStorage` 类（接口与本地存储完全相同），`R2_ACCOUNT_ID` 环境变量存在时自动切换，代码不变。

### 6.3 存储路径规范（R2）

| 文件类型 | 存储路径 |
|----------|---------|
| 产品封面图 | `/products/{productId}/cover.{ext}` |
| 产品相册图 | `/products/{productId}/gallery/{n}.{ext}` |
| 产品视频 | `/products/{productId}/video.{ext}` |
| 文章封面图 | `/articles/{articleId}/cover.{ext}` |

### 6.4 删除联动

删除产品或文章时，同步调用 `deleteR2Objects()` 清理 R2 中对应的文件，防止废弃文件堆积。本地 mock 模式下此操作静默跳过。

## 7. 数据库结构

### 7.1 切换机制

- `SUPABASE_URL` + `SUPABASE_SERVICE_ROLE_KEY` 同时设置 → 所有 `getXxxRepo()` 返回 Supabase 实现
- 未设置 → 返回内存 mock（`globalThis.__kxtjMockStore__`，服务重启数据清空）

```
我在开发一个外贸独立站，Next.js 全栈项目，二手工程机械出口业务。

【技术架构】
- 框架：Next.js 16.2.1，React 19.2.4，App Router
- 样式：Tailwind CSS 4，tailwind-merge
- UI组件：@base-ui/react（不是 Radix UI）
- 多语言：next-intl 4.x，支持 en/zh/ar/ru/es，路由结构 /[locale]/...
- 状态管理：Zustand 5.x
- 表单验证：Zod 4.x
- 文件存储：本地开发用 public/uploads/，生产用 Cloudflare R2（@aws-sdk/client-s3，S3 兼容）
- 数据库：SUPABASE_URL + SUPABASE_SERVICE_ROLE_KEY 存在时用 Supabase；否则用内存 mock
  - 接口定义：src/lib/data/repository.ts
  - Supabase 实现：src/lib/data/supabase-repository.ts（已完成）
  - 建表 SQL：docs/supabase-schema.sql
- 认证：jose + bcryptjs，通过 hasAdminSession() 检查 admin_token cookie
- 富文本：@tiptap/react 已安装，待接入后台编辑器
- 部署：Vercel（待上线）

【环境变量切换机制】
- SUPABASE_URL + SUPABASE_SERVICE_ROLE_KEY 同时存在 → 用 Supabase，否则用内存 mock
- R2_ACCOUNT_ID 存在 → 用 Cloudflare R2，否则用本地 public/uploads
- 上线只需填环境变量，不改代码

【后台登录】
- 地址：/admin/login
- 邮箱：admin@kxtj.com / 密码：admin123456（开发测试账号，接 Supabase 后需重建）

【已完成的 API（全部实现）】
公开接口：
- POST/GET /api/inquiries（含速率限制）
- POST /api/inquiries/[id]/read
- GET /api/products，GET /api/products/[slug]
- GET /api/articles，GET /api/articles/[slug]
- GET /api/site-settings（60s ISR 缓存）
- GET/POST /api/page-content（?pageId=home&locale=zh）

后台接口（需 admin_token cookie）：
- POST /api/admin/login，POST /api/admin/logout
- GET /api/admin/dashboard（询盘数、未读数、产品数、草稿数）
- GET/POST /api/admin/products
- GET/PATCH/DELETE /api/admin/products/[id]
- POST /api/admin/uploads
- PATCH /api/admin/inquiries/[id]（status / isRead）
- GET /api/admin/categories（从产品派生分类统计）
- GET/POST /api/admin/articles
- GET/PATCH/DELETE /api/admin/articles/[id]
- GET/PUT /api/admin/settings

【前台 Hooks】
- useSiteSettings：读取站点设置（站名、Logo、联系方式、社交、版权），带全局内存缓存
- usePageContent：读取页面 CMS 内容
- useInquirySubmit：询盘提交
- useProductCatalog：产品目录

【代码结构原则】
- 新建 API 时，业务逻辑放 src/lib/services/，route 只做参数解析和响应
- 前台 fetch 调用统一放 src/lib/api/ 或对应 hook，不散落在组件里
- 单文件不超过 200 行

【待办文档】
- 全局分析与待办：docs/11_项目全局分析与待办总表.md
```
